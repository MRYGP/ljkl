# AGENTS.md — 本仓库的 agent 操作规约

> 给 Codex / Claude / 任何 coding agent 读。人也可读。
> 维护本仓库前，先读 `00-宪章.md`（世界观+铁律），再读 `坐标系.md`（分类标准）。这两份是不可违反的上位约束。

## 这个仓库是什么
纯学术研究档案，研究「伟大从哪些结构位置长出」并预测当下的结构空位。**不是产品仓库，不做变现，不设 KPI。** 详见宪章。

## 目录结构
```

├── README.md            人看的入口与索引
├── AGENTS.md            本文件，agent 操作规约

└── 格子档案\
    ├── _模板.md         新格子从此复制
    └── CELL-NNN-*.md    一格一文件，带 YAML frontmatter
```

## 三条不可违反的铁律（违反即拒绝操作）
1. **分析单元是踏脚石/被锁的需求，不是产品。** 收到「拆某产品怎么赢」的任务 → 那是 SK（D:\sk）的活，不在此处理。
2. **无墓碑不入档。** 新增/补全格子档案时，`## 墓碑` 栏为空则该格标记为「不合格」，不得作为预测依据。
3. **不功利。** 不得在任何文件引入变现目标、增长 KPI、回报期限。

## 常见任务怎么做

### 新增一个格子
1. 复制 `格子档案\_模板.md` → `格子档案\CELL-NNN-<踏脚石>-<层>-<象限>.md`（NNN 取当前最大+1）
2. 填满 frontmatter（layer/region/demand_quadrant/redline/status/evidence/updated）
3. 正文必填三栏：踏脚石、活公司、**墓碑**。墓碑空 → status 标 `open` 并在文末注明欠账。
4. `redline: true` 时：标题前加 🔴，正文只记机制、删除一切可复制的操作细节。

### 新增一条预测
追加到 `空格子-预测库.md`，严格按文件顶部「写法规约」。墓碑/生死比缺失的预测标为半成品，进「待补墓碑清单」。

### 对账
到期预测：不改原文，在其下追加 `> 对账 YYYY-MM：命中/落空/部分——原因`。

### 命名铁律（沿用 SK）
- 文件名**永不带版本号**。版本写在文件内首行 `> 版本：vX.Y · YYYY-MM-DD`。
- 例外：研究报告/深度底稿类允许版本号。

## 编辑纪律
- 只改本次任务直接触发的部分，不重写无关段落、不调格式、不删注释。
- 发现断链/错路径，先在 commit message 或回复里告知人，不自行顺手改方向性内容。
- frontmatter 是机器可解析入口，字段名不要随意改；要改先改 `_模板.md` 并同步存量文件。

## 与 SK 的关系
SK 在 `D:\sk`，是独立项目。本项目左下象限（巨头不屑的 L3 垂直）与 SK 腹地重叠——**这里只做结构记录，落地一律交还 SK，不在本仓库推产品、算现金流。**

## Git 提交规约
仓库已发布：https://github.com/MRYGP/ljkl
- **每完成一个 TASK 或一个格子，提交一次**，不要攒一堆再提交。提交后 push 到 origin main。
- commit message 格式：`<类型>: <一句话>`。类型：`feat`(新格子/新预测) / `fix`(修正判断或象限) / `docs`(改宪章/坐标系/README/AGENTS) / `chore`(维护)。
  - 例：`feat: CELL-001 补墓碑，生死比 3/5，证据升 B`
  - 例：`fix: CELL-001 象限由左下改左上，墓碑死法指向技术锁`
- **改上位文件（00-宪章.md / 坐标系.md / AGENTS.md）必须单独成一个 commit**，message 写明改动理由——这类文件是全项目的语言，改动要可追溯、可回滚，不与内容改动混在一个提交里。
